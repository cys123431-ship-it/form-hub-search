# Form Hub MVP 기능명세서

## 1. 문서 목적
- 무료 양식과 공개 채용 지원서 양식을 한곳에서 찾고 비교할 수 있는 검색형 앱의 MVP 범위를 정의한다.
- 본 명세서는 `/home/joseph/바탕화면/개발/first.md`의 우선순위와 가드레일을 따른다.
- 이번 MVP는 별도 신규 앱 `form-hub`를 구축하는 방향을 기준으로 작성한다.

## 2. 우선순위와 제약

### 2.1 제품 우선순위
- `Security > Correctness > Reliability > Maintainability > Speed`
- 검색 품질보다 먼저 수집 적법성과 데이터 정합성을 보장한다.

### 2.2 법적/운영 제약
- `MUST` 크롤링 허용 범위가 확인된 공개 페이지와 공개 첨부파일만 수집한다.
- `MUST` 저작권 또는 이용약관상 재배포가 금지된 파일은 원본 저장/재배포하지 않는다.
- `MUST` 결과 화면에는 원문 출처 URL을 항상 노출한다.
- `MUST` 차단된 소스는 수집 대상에서 제외한다.
- `SHOULD` 첨부파일 자체보다 메타데이터, 요약, 허용된 미리보기를 우선 제공한다.

### 2.3 MVP 범위 현실화
- `MUST` HTML 본문과 PDF 기반 공개 양식을 우선 지원한다.
- `SHOULD` `DOCX`, `HWPX`는 추출 어댑터 구조만 먼저 만든다.
- `MAY` `HWP` 바이너리는 MVP에서 미지원하고 메타데이터만 저장한다.

## 3. 문제 정의
- 무료 양식은 여러 사이트에 흩어져 있고 분류 기준이 제각각이다.
- 채용 지원서는 기업/기관별로 공고 페이지, 첨부파일, 외부 지원 폼이 나뉘어 있어 한 번에 보기 어렵다.
- 사용자는 "이력서", "공기업 자소서", "산업은행 지원서"처럼 복합 조건으로 빠르게 찾고 싶다.

## 4. 목표와 비목표

### 4.1 목표
- 태그 기반으로 무료 양식을 검색한다.
- 여러 태그를 `AND`/`OR`로 조합해 필터링한다.
- 공개 채용 지원서 양식을 기업/기관/채용유형 기준으로 묶어 보여준다.
- 결과 목록에서 제목, 태그, 출처, 파일형식, 요약, 미리보기를 확인한다.

### 4.2 비목표
- 유료 양식 판매 플랫폼 통합
- 로그인 기반 개인화 추천
- 문서 편집기/작성기 기능
- 전체 웹 대상 범용 검색엔진
- 무제한 파일 재호스팅

## 5. 핵심 사용자 시나리오

### 5.1 일반 양식 검색
1. 사용자가 `근로계약서`, `프리랜서`, `디자인` 같은 태그를 선택한다.
2. 검색 모드를 `교집합` 또는 `합집합`으로 정한다.
3. 앱이 조건에 맞는 양식 목록과 요약 미리보기를 반환한다.
4. 사용자가 상세 보기에서 출처 링크와 첨부파일 링크를 확인한다.

### 5.2 채용 지원서 묶음 검색
1. 사용자가 `산업은행`, `기업은행`, `공기업`, `자소서` 같은 키워드 또는 태그를 입력한다.
2. 앱이 관련 공고/지원서/자기소개서 양식 후보를 묶어서 보여준다.
3. 사용자는 기업명, 채용 시즌, 파일 유형, 마감 여부로 다시 좁힌다.
4. 상세 보기에서 원문 공고와 지원 양식 링크를 한 화면에서 본다.

## 6. 사용자 유형
- 구직자: 기업/기관별 지원서 양식을 모아서 보고 싶다.
- 실무자/프리랜서: 계약서, 제안서, 증빙 양식을 빠르게 찾고 싶다.
- 운영자: 허용된 소스를 등록하고 수집 상태를 점검하고 싶다.

## 7. MVP 기능 범위

### 7.1 수집 기능
- `MUST` 등록된 소스 목록을 주기적으로 순회한다.
- `MUST` 소스별 파서가 제목, 본문, 출처 URL, 첨부파일 URL, 게시일을 추출한다.
- `MUST` 수집 시간, 소스 상태, 실패 이유를 기록한다.
- `MUST` 중복 문서를 감지한다.
  - 기본 키 후보: `canonical_url`, `attachment_url`, `content_hash`
- `SHOULD` 같은 양식의 재업로드본은 하나의 대표 문서 아래 버전/출처만 묶는다.

### 7.2 텍스트 추출 기능
- `MUST` HTML 본문은 정제 후 텍스트로 저장한다.
- `MUST` PDF는 텍스트 레이어가 있을 때 본문 추출을 시도한다.
- `SHOULD` 파일 추출 실패 시 상태값을 남기고 메타데이터만 노출한다.
- `MAY` 이미지형 PDF/OCR은 차기 단계로 미룬다.

### 7.3 자동 분류 기능
- `MUST` 규칙 기반 태그 분류를 지원한다.
- `MUST` 문서 유형, 주제, 채용 관련 메타태그를 분리 저장한다.
- `SHOULD` 태그는 다음 3계층으로 관리한다.
  - 문서 유형: `이력서`, `자소서`, `근로계약서`, `제안서`, `동의서`
  - 도메인: `채용`, `노무`, `법률`, `부동산`, `디자인`, `행정`
  - 기관/속성: `산업은행`, `기업은행`, `공기업`, `대기업`, `신입`, `인턴`
- `SHOULD` 규칙 기반 분류 결과에 신뢰도 점수를 저장한다.
- `MAY` LLM 보조 분류는 운영자 승인 후 추가한다.

### 7.4 검색 기능
- `MUST` 키워드 검색을 지원한다.
- `MUST` 태그 다중 선택과 `AND`/`OR` 모드를 지원한다.
- `MUST` 결과 정렬 기준을 제공한다.
  - 최신순
  - 관련도순
  - 출처 신뢰도순
- `MUST` 결과 목록에 다음 필드를 보여준다.
  - 제목
  - 대표 태그
  - 출처명
  - 게시일 또는 수집일
  - 파일 형식
  - 짧은 요약
  - 미리보기 가능 여부
- `SHOULD` 기업/기관명 필터와 채용 유형 필터를 분리 제공한다.

### 7.5 상세/미리보기 기능
- `MUST` 상세 화면에서 본문 요약, 태그, 출처 링크, 첨부파일 링크를 제공한다.
- `MUST` 미리보기는 저장이 허용된 텍스트/썸네일만 노출한다.
- `MUST` 파일 재배포가 금지된 경우 원문 링크 이동만 제공한다.

### 7.6 운영 기능
- `MUST` 소스 등록 정보에 수집 허용 여부, 파서 유형, 주기, 타임아웃을 저장한다.
- `MUST` 최근 수집 성공/실패 상태를 운영자가 확인할 수 있어야 한다.
- `SHOULD` 태그 사전과 키워드 규칙을 운영자가 수정할 수 있어야 한다.

## 8. 상세 요구사항

### 8.1 검색 규칙
- 키워드는 제목, 본문 추출 텍스트, 태그, 기관명, 출처명을 대상으로 매칭한다.
- `AND` 모드는 선택 태그 전체를 만족해야 한다.
- `OR` 모드는 선택 태그 중 하나 이상 일치하면 된다.
- 복수 키워드 입력 시 기본 동작은 `AND`로 처리한다.
- 기관명과 일반 태그를 함께 입력한 경우:
  - 예: `산업은행 + 자소서`
  - 기관 태그와 문서 유형 태그를 동시에 만족하는 결과를 우선 노출한다.

### 8.2 중복/품질 규칙
- 동일 `canonical_url`이면 같은 문서로 본다.
- URL이 달라도 첨부파일 해시가 같으면 중복 후보로 본다.
- 제목만 같고 내용이 다른 경우는 별도 문서로 저장한다.
- 본문 길이가 너무 짧거나 태그가 하나도 나오지 않으면 `검수 필요` 상태로 표시한다.

### 8.3 요약 규칙
- 요약은 본문 앞부분 그대로 노출하지 않고 정제된 핵심 문장 2~3개로 만든다.
- 개인정보나 지원자 예시 데이터가 포함되면 제거 후 저장한다.

## 9. 비기능 요구사항

### 9.1 보안
- `MUST` 환경변수로만 시크릿을 주입한다.
- `MUST` 입력 파라미터를 검증한다.
- `MUST` DB 접근은 파라미터 바인딩 또는 ORM으로 처리한다.
- `MUST` 외부 요청에 타임아웃을 둔다.

### 9.2 신뢰성
- `MUST` 크롤링 실패는 문서 단위가 아니라 작업 단위 로그로 남긴다.
- `MUST` 일시적 네트워크 실패에 재시도를 적용한다.
- `SHOULD` 파서 실패가 다른 소스 수집을 막지 않도록 격리한다.

### 9.3 유지보수성
- `MUST` 소스별 파서는 어댑터 단위로 분리한다.
- `MUST` 분류 로직은 순수 함수 중심으로 분리한다.
- `MUST` 검색 API와 크롤링 작업 로직을 분리한다.

## 10. MVP 정보구조

### 10.1 사용자 화면
- 홈: 검색창, 인기 태그, 최근 수집 문서
- 검색 결과: 태그 필터, `AND/OR` 토글, 결과 리스트
- 상세 보기: 요약, 태그, 출처, 첨부 링크, 유사 문서

### 10.2 운영 화면
- 소스 목록
- 수집 실행 이력
- 검수 필요 문서 목록

## 11. 수용 기준
- 사용자가 태그 2개 이상을 선택하고 `AND/OR` 검색 결과 차이를 확인할 수 있다.
- `산업은행`, `기업은행`, `공기업`, `자소서` 같은 복합 검색이 동작한다.
- 수집된 문서는 출처 링크와 함께 표시된다.
- 중복 문서가 단일 결과로 정리된다.
- 파서 실패 문서는 서비스 오류가 아니라 상태값으로 관리된다.

## 12. 단계별 산출물 연결

### 12.1 2단계 DB 스키마 설계 입력값
- 문서
- 문서 본문 추출 결과
- 태그/태그 매핑
- 소스 설정
- 수집 작업 이력
- 중복 그룹

### 12.2 3단계 API/크롤러 설계 입력값
- 소스 파서 인터페이스
- 분류 파이프라인
- 검색 쿼리 계약
- 운영 상태 조회 API

### 12.3 4단계 프로토타입 범위
- 실제 크롤링은 허용된 샘플 소스 또는 로컬 목데이터 기준
- 태그 검색과 상세 미리보기가 동작하는 UI
- 향후 실제 수집기로 교체 가능한 구조 유지

## 13. 오픈 이슈
- `HWP/HWPX` 추출 범위를 MVP에 포함할지 여부
- 실제 운영 대상 소스 목록과 이용약관 검토 방식
- 기업 채용 양식에서 외부 지원 시스템 링크만 제공할지, 첨부파일만 수집할지 여부
